# Mishka Den — Project Documentation

**Version:** 0.6.0 | **License:** MIT | **PHP:** >=8.4

A family hub web app — the den mother for your family. First real-world dogfood of the [karhu](https://github.com/bjornbasar/karhu) PHP microframework.

This file is the top-level overview. Detail lives in `docs/`:

- **[docs/SCHEMA.md](docs/SCHEMA.md)** — full database schema, every table, design notes per release
- **[docs/ROUTES.md](docs/ROUTES.md)** — full route table grouped by feature
- **[docs/CALENDAR.md](docs/CALENDAR.md)** — v0.3 calendar design (time model, month grid, optimistic concurrency, recurrence, single-occurrence editing, iCal feed)
- **[docs/CHORES.md](docs/CHORES.md)** — v0.4 chores design (overdue/time model, live points tally + credit rule, recurring-chore generation + round-robin rotation)
- **[docs/ACCOUNT.md](docs/ACCOUNT.md)** — v0.5 account lifecycle (profile, password change, reset, verify, household lifecycle, session revocation, rate limit, threat model)

## Top-level design decisions

(Detail in `docs/SCHEMA.md`, `docs/CALENDAR.md`, and the per-component class docstrings.)

1. **Email as identifier, integer PK for FK targets** (v0.1). karhu's `UserRepositoryInterface` is keyed on an opaque string; mishka puts the email in that slot, and `users.id` is the canonical FK target.
2. **Lowercase-on-write email policy** (v0.1). Single UNIQUE constraint, no functional index.
3. **Race-free first-user-admin via sentinel row** (v0.1). Atomic `UPDATE` claims the seeded admin slot.
4. **Timing-attack-safe login** (v0.1). One `password_verify` call per branch, including a dummy hash for the unknown-email path.
5. **Logout clears the session cookie** (v0.1). `setcookie()` with expired timestamp after `Session::destroy()`.
6. **Controllers read both JSON and form-urlencoded bodies** (v0.1). Test harness sends JSON; browsers send urlencoded.
7. **N:M user → household** (v0.2). Divorced parents, foster carers, live-in nannies are 10-15% of real users.
8. **Race-free creator-as-owner; no auto-promote-first-joiner** (v0.2). Single-transaction create + owner-membership insert.
9. **Controller-level no-household guards** (v0.2). Karhu's middleware runs before route matching; path-string exemptions are fragile.
10. **Last-selected active household in `user_preferences`** (v0.2). Restored on login.
11. **Stale-session self-heal via ForbiddenException(redirectTo)** (v0.2). Kicked users land on /household/setup, not a 403.
12. **Multi-tab CSRF "just works"** (karhu v0.1.1). Token rotates only on session rotation, not every POST.
13. **Local-time + IANA timezone storage for events** (v0.3.0). UTC TIMESTAMPTZ drifts under DST. Recurr expands in the event's tz.
14. **Schema additive-only via `CREATE TABLE IF NOT EXISTS`** (v0.3.0 includes inert `rrule` + `series_event_id` columns) so v0.3.1 doesn't need an ALTER. **v0.6.6 exception (see #47):** adding columns to a table that already exists in prod cannot be expressed via `CREATE TABLE IF NOT EXISTS`; v0.6.6 introduces the first ALTER inside a fenced PG-only block at the end of `db/schema.sql`.
15. **Optimistic concurrency on event edits** (v0.3.0). `_expected_updated_at` hidden field; 409 + stale-data partial on mismatch.
16. **POST whitelist** (v0.3.0). Calendar controller never accepts `series_event_id`, `timezone`, `created_by`, or system columns from form input.
17. **Pattern B for single-occurrence editing** (v0.3.1). `event_exceptions(event_id, original_starts_at, override_event_id NULL)` with RFC 5545 RECURRENCE-ID semantics in mind for the future iCal feed.
18. **Two-step DELETE for dropping overrides** (v0.3.1). FK CASCADE points `event_exceptions → events`, so deleting an exception row does NOT delete the override Event. `EventExceptionRepository::dropAllForEvent` deletes the override events first (CASCADE wipes the exception rows), then deletes the remaining cancellation rows.
19. **Cascade-on-series-edit with confirmation dialogs** (v0.3.1). Clean time-shifts cascade override `original_starts_at` by the same delta; structural rrule/all_day changes drop overrides with a list of what's affected. `_expected_exception_count` hidden field protects the dialog flow against another tab adding/removing exceptions mid-dialog.
20. **Defensive `series_event_id IS NULL` + `rrule IS NULL` filter** (v0.3.1) in `EventRepository::findInRangeForHousehold`. Override events would otherwise double-render through the one-off branch; recurring series would otherwise leak through the same branch alongside the RangeExpander.
21. **SHA-256 hashed iCal tokens with cap at 3** (v0.3.2). Raw 64-hex token shown to the user once; only the hash persists. Cap-at-3 auto-revokes the oldest active row on the 4th generate — bounded leak surface without forcing manual cleanup. `last_used_at` exposed in settings as a leak-detection signal.
22. **sabre/vobject not eluceo/ical for iCal serialisation** (v0.3.2). eluceo/ical 2.x cannot emit `RECURRENCE-ID`; overrides need it. sabre/vobject also parses iCal — door open for v0.5+ "subscribe to external calendar".
23. **Layered token-leak defences** (v0.3.2). `Referrer-Policy: no-referrer` on feed responses + `<meta name="referrer">` on the post-generate page + Caddy log-path redaction documented in INFRASTRUCTURE.md.
24. **Live points tally, no ledger** (v0.4.0). `pointsTallyForHousehold` sums completed chores by `COALESCE(completed_by, assigned_to)` — the doer — driven off current `household_members`. `ORDER BY MIN(joined_at)` so PostgreSQL's GROUP BY rule holds. Documented limitation: editing/deleting a completed chore or removing a member shifts the tally; a durable ledger is the v0.4.2+ path.
25. **Chores inert columns ship without a DB FK** (v0.4.0). `chores.schedule_id` is a bare `INTEGER` (no `REFERENCES`) because its v0.4.1 target table doesn't exist yet and the no-ALTER convention can't add the constraint later — integrity is app-enforced. No defensive `schedule_id IS NULL` list filter (templates live in a separate table, so v0.4.1 instances are first-class list rows, unlike the calendar's double-render risk).
26. **Overdue computed in PHP against the chore's own timezone** (v0.4.0), never SQL `NOW()`. NULL due = never overdue. Same wall-clock + IANA model as events; the predicate is duplicated in `ChoresController` and `HomeController` (no shared base).
27. **Recurring chores = template + generated instances** (v0.4.1). `chore_schedules` is the template; occurrences are generated as ordinary `chores` rows (Tody-style) so completion/points/overdue reuse v0.4.0. Generated lazily on view, bounded to a 14-day rolling horizon via a `generated_through` watermark + a 60-row cap — because recurr always expands from the anchor, an unclamped limit would either silently generate zero rows or explode.
28. **Rotation cursor is a durable id, not an index** (v0.4.1). `last_assigned_user_id` + a pure-function next-assignee survive member renumbering and concurrent lazy generation; the cursor advances only alongside a successful insert. `assignment_mode='fixed'` pins instead of rotating.
29. **`ChoreSchedulesController` scanned before `ChoresController`** (v0.4.1). The router matches sequentially, so the static `/chores/schedules` routes must precede `/chores/{id}`.
30. **Schedule edit refreshes upcoming; delete is app-coordinated** (v0.4.1). Edit deletes future-open occurrences + rewinds the watermark; delete drops open + detaches completed (`schedule_id` → NULL) because `chores.schedule_id` has no FK cascade.
31. **Durable points ledger replaces the live tally** (v0.4.2). `markDone` writes a `chore_points_ledger` row iff the guarded UPDATE transitioned the chore (`run()===1`), one UTC timestamp to both rows; `reopen` deletes it; no `UNIQUE(chore_id)` (reopen→recomplete). Editing/deleting a completed chore no longer mutates history (chore_id/credited_user_id SET NULL). Idempotent `NOT EXISTS` backfill in `schema.sql`.
32. **Weekly leaderboard windowed in PHP, compared in UTC** (v0.4.2). Monday 00:00 household tz → UTC string bound vs the ledger's UTC `completed_at`, so PG (TIMESTAMPTZ) and SQLite (TEXT) agree. Ranked `week_points DESC, MIN(joined_at)`.
33. **Pause via flag-table; participant pools via subset-table** (v0.4.2). `chore_schedules` can't gain columns (no ALTER), so pause = presence in `chore_schedule_pauses` (skipped in `generateForHousehold`, never inside `generateForSchedule`, so the watermark can't drift); a pool = rows in `chore_schedule_participants` (rotation cycles `listMembers ∩ pool`, empty intersection → unassigned). Both new tables carry real FKs.
34. **DST-safe week arithmetic centralised in `WeekWindow`** (v0.4.3). Monday 00:00 NZDT and Monday 00:00 NZST are NOT 168 UTC-hours apart (169 at the end of DST, 167 at the start), so naive `−7d` on a UTC string drifts by an hour across every transition and silently breaks streak walks. `WeekWindow` does every step in household tz via `->modify('-1 week')->setTime(0, 0, 0)`, converting to UTC only for the string representation. Both controllers + `Achievements` route through it.
35. **Badges + streaks are stateless** (v0.4.3). Derived per-render from the ledger; no `member_badges` table, no `earned_at` history. Badge criteria are pure functions over a stats array, returned from a method (not `const` — PHP rejects closures in constant expressions). Presentation (emoji + title) registered as the `badge_meta` Twig global, mirroring `brand`; the service never sees emoji.
36. **Host-header-injection guard via boot-required `APP_URL`** (v0.5.0). The unauthenticated `/password-reset` endpoint would otherwise be a phishing factory: forge `Host: evil.com`, request a reset for a victim, and SMTP-deliver `https://evil.com/password-reset/<token>`. Fix: `App\Mail\UrlBuilder` reads only `$_ENV['APP_URL']`; `public/index.php` validates the env var at boot (must match `^https?://`). No code path reads `$request->header('host')` for emailed URLs (B1).
37. **Token pattern reused** (v0.5.0). `email_verification_tokens` and `password_reset_tokens` mirror `ical_feed_tokens` byte-for-byte: 32 random bytes → 64-char hex shown ONCE → SHA-256 stored as `token_hash` with a UNIQUE index. One canonical token shape across the app reduces the bug surface.
38. **Atomic single-use redeem via guarded UPDATE** (v0.5.0). Both `password_reset_tokens` and `email_verification_tokens` redeem via `UPDATE … SET used_at = :now WHERE id AND used_at IS NULL AND expires_at > :now` and gate the post-redeem write on `rowCount === 1`. Two concurrent POSTs of the same token cannot both succeed (B6). Expiry math is in PHP via `gmdate('Y-m-d H:i:s')` because SQLite doesn't translate `NOW() + INTERVAL` (B3).
39. **Owner-transfer atomicity with FOR UPDATE on PG** (v0.5.0). The non-unique partial index `idx_household_members_role` can't enforce single-owner. `transferOwnership` opens a txn, runs `SELECT user_id, role FROM household_members WHERE household_id=:h AND user_id IN (:old,:new) FOR UPDATE` (no-op on SQLite — single-writer makes the lock implicit; PG-only syntax detected via PDO driver name), re-verifies inside the lock, then promote-first / demote-second with guarded UPDATEs. Two-owner intermediate state is degenerate-OK; zero-owner would be degenerate-bad and the rowCount check blocks it (B5 + round-4 BL-3).
40. **Session revocation via separate stamp table** (v0.5.0). `users` can't gain a `password_changed_at` column (additive-only schema), so `user_password_changes (user_id PK, password_changed_at)` is the stamp. `SessionRevocationGuard` middleware compares `Session::get('auth_time')` to the stamp; 4-permutation predicate handles missing-state combinations (legacy session pre-v0.5 = grandfather pass per user decision U-1, BL-1). The pinned-`$now` invariant (BL-2) — caller pins one `gmdate()` call and passes the SAME string to `updatePassword`'s stamp and `Session::set('auth_time')` — prevents the user from self-revoking on their own password change.
41. **App-layer rate limit via simple counter table** (v0.5.0). `email_send_attempts (kind CHECK IN ('password_reset_request','verify_resend'), ip_address NULL, user_id NULL, attempted_at)` records every issuance attempt; the controller does `countRecentByIp/ByUser(kind, key, 10min)` before issuing. Two independent buckets (5/10min/IP for reset, 3/10min/user for resend) so one bucket overflowing doesn't lock the other. Unbounded growth is fine at family-scale; a future prune job can DELETE old rows (H4).
42. **Soft-banner email verification with always-quiet copy** (v0.5.0). No app features are gated on `email_verified_at`. The layout banner reads "Please verify your email — [Resend]" regardless of SMTP outcome (user decision U-3); `email_verification_tokens.sent_at` distinguishes for ops, never for users. Family-friendly tone, no scary "we couldn't send" copy.
43. **Always-200 + 1.5s timing floor + dummy argon2id verify on miss** (v0.5.0). The hit path takes ~100–500ms (issue + Twig render + SMTP); the miss path was ~5ms. Three layers close enumeration: (a) identical body for hit/miss; (b) `usleep(max(0, 1_500_000 - elapsed_us))` floor; (c) on miss, run a throwaway `PasswordHasher::verify` against a precomputed dummy hash so the argon2id baseline cost lands on miss too (round-4 H-4 — defence in depth for SMTP-timeout edge cases that blow past the floor on hit). Tests use a 50ms floor for speed; `PasswordResetTimingTest` exercises the real 1.5s floor.
44. **`Referrer-Policy: no-referrer` on token-bearing routes** (v0.5.0, round-4 H-5). Both `/password-reset/{token}` GET responses (form + invalid page) and `/verify-email/{token}` GET set the header, preventing the raw token from leaking via `Referer` if the user clicks an external link from those pages.
45. **PWA manifest as the iOS Web Push unblocker** (v0.6.3). iOS Safari only delivers Web Push when the site is installed as a real PWA (`display: standalone`) — `<link rel="manifest">` + an `apple-touch-icon` + the `apple-mobile-web-app-*` meta tags are what flip "Add to Home Screen" from bookmark to PWA. Manifest `id: "/"` is locked forever (changing it orphans every existing install). Status bar style is `black` not `black-translucent`, because the latter renders content under the iPhone notch and we don't carry `viewport-fit=cover` + `env(safe-area-inset-top)` CSS retrofits. Manifest `theme_color` / `background_color` are snapshotted at install time; a 3-way drift test (`tests/View/ManifestTest.php`) keeps CSS vars, meta tag, and manifest field aligned. Icons are Lanczos upscales from a single 192×192 source — replaceable later without a version bump because static-asset bitmaps are versionless.
46. **Progressive-enhancement hamburger as mishka's first explicit ARIA disclosure** (v0.6.4). Pre-v0.6.4 mishka had implicit ARIA via the household-switcher's `<details>`/`<summary>`, but no author-supplied ARIA attributes. The hamburger introduces `<button aria-expanded aria-controls aria-label>` + sibling-selector CSS (`.nav-toggle[aria-expanded="true"] + nav`). Three coordinated parts make the FOUC-free + no-JS-safe behaviour work: (a) a pre-paint `<script>` in `<head>` adds `.js-nav` to `<html>` before first paint so the @media rule knows JS is enhanced — without it, mobile users would see one paint frame of the 568px overflowing nav before the body-end script collapses it. (b) `.nav-toggle { display: none }` outside @media + `.js-nav header.top .nav-toggle { display: inline-block }` inside @media (max-width: 700px) means no-JS users (no `.js-nav` class) never see a button that wouldn't work; they fall back to the pre-v0.6.4 overflowing flex row — same UX as before, no lockout. (c) A `matchMedia('(max-width: 700px)')` listener resets `aria-expanded='false'` when the viewport crosses out (iPad rotation) so SR users don't hear "expanded" on a no-longer-meaningful state. Touch target is 48px (Material Design recommendation, not Apple HIG — 48 exceeds Apple's 44pt floor either way). The glyph `☰` is wrapped in `<span aria-hidden="true">` so screen readers don't announce "trigram for heaven". `prefers-reduced-motion` is not gated because there is no animation. Forced-colors mode supplies `ButtonText` / `CanvasText` fallbacks; print stylesheet hides the entire nav + footer + flash banners. The bell link's `font-size` moved from an inline `style` attribute to a CSS rule (`.nav-bell`) so the drawer's override can win via cascade order (inline always beats class selectors; matched specificity + later rule does not).
47. **Creation-time push categories + mishka's first schema ALTER** (v0.6.6). Two new push kinds — `new_chore_assigned` (POST /chores) and `new_event` (POST /calendar/events, fan-out to all members except creator) — fire as synchronous side effects of the controller POST handlers, not on a push:scan cron tick. **Inline-not-helper**: each controller has a private `enqueueXxxPush()` method copying `PushScanCommand`'s "open txn if not in one, claim+push, commit, throw on failure" B3 semantics. No shared `PushDispatcher` class until a 5th callsite appears. **Scope is creation-only**: edits (POST /chores/{id}, /calendar/events/{id}) don't push, override-occurrence saves don't push, `ChoreScheduleGenerator`-generated recurring instances don't push (they go via `ChoreRepository::createGenerated`, not `handleCreate`). **Defaults are opt-out** — both new prefs default `TRUE`. **First explicit ALTER in mishka's schema** (revises decision #14): pre-v0.6.6, schema changes were strictly additive via `CREATE TABLE IF NOT EXISTS` on new tables; v0.6.6 needed to add 2 columns to `user_notification_prefs` and extend the CHECK predicate on `notification_dispatches.kind`, both impossible via the IF-NOT-EXISTS pattern once the tables exist in prod. The fix: a fenced `-- BEGIN PG_ONLY ... -- END PG_ONLY` block at end-of-`schema.sql` containing 4 idempotent ALTERs wrapped in `BEGIN; ... COMMIT;` for atomic partial-failure rollback. 69. **Family "stay logged in" — 30-day session lifetime** (v0.7.7). Closes the DOCS #65 v0.7.4+ candidate flagged in the v0.7.3 persistent-session-storage postmortem. Pre-v0.7.7 sessions used PHP's language defaults: `session.gc_maxlifetime = 1440` (24 minutes idle before server-side GC) + `session.cookie_lifetime = 0` (browser-session cookie, dies when the tab/browser closes). For a family app installed as a PWA on tablet or phone, this meant users had to re-authenticate every time they picked up the device — jarring UX for a personal household hub, and misaligned with the v0.7.3 investment in persistent session storage (server-side files survive container recreates, but the browser cookie died as soon as the tab closed anyway). **Fix**: paired config bumps, both to 2,592,000 seconds (30 days = 30 × 24 × 60 × 60). Server side: extended `/usr/local/etc/php/conf.d/mishka-sessions.ini` (the v0.7.3 file that pinned `session.save_path`) to also set `session.gc_maxlifetime = 2592000` — same file, two directives, one runtime-stage `RUN` in the Dockerfile. Client side: `new Session(['lifetime' => 2592000])` in `public/bootstrap.php:314` overrides karhu Session middleware's default `'lifetime' => 0` cookie param. Karhu Session's ctor merges the passed array into its defaults + calls `session_set_cookie_params()` before `session_start()` — so every new session (or regenerate) writes a `PHPSESSID` cookie with `Max-Age=2592000; expires=<UTC now+30d>`. **Semantic**: 30 days from LOGIN (or from `Session::regenerate()`, which fires on login + password-change), NOT "30 days from last activity". `session_set_cookie_params()` is called at the start of the session; browsers hold the cookie until the recorded `expires` regardless of subsequent requests. Simple mental model; matches how bank apps and most consumer PWAs work. **Server-side alignment**: `session.gc_maxlifetime` is the "idle time before GC eligible" threshold — PHP's random-probability GC (`gc_probability=1 / gc_divisor=1000` = 0.1% per request in the Debian default) prunes session files that haven't been read for >30 days. Files created in the v0.7.3 named volume `/var/lib/mishka/sessions` now stay pruneable-only after 30 days of no read, matching the browser cookie's max lifetime. If a user is active daily, their session file's mtime updates on every read, and the file never approaches GC eligibility. **Why safe** (three defence layers already in place, v0.7.0-v0.7.4): (1) `/me/sessions` UI (v0.7.0) lets any user revoke per-device on demand — "sign out everywhere" is one tap. 66. **Client-side double-submit prevention** (v0.7.4 hotfix). Second distinct class of mobile-CSRF-403 that survived v0.7.3's persistent-session-storage fix. **Signal**: post-v0.7.3 Ayula nginx access log (`/var/log/nginx/access_upstream.log`) still showed CSRF 403s from Android Chrome / PWA (mobile + tablet). But now the pattern was different: `POST /login → 303` (successful login) followed **~1 second later** by a second `POST /login → 403` from the same IP+UA. Six user sessions in the ~1h window post-deploy showed the same shape; Windows Chrome from the operator's IP never showed the follow-up 403. **Root cause**: `AuthController::establishSession` (called on successful login) invokes `Session::regenerate()` then wipes `$_SESSION`, then karhu's `Csrf::regenerate()` writes a fresh `_csrf_token` to the NEW session. The `_csrf_token` embedded in the pre-submit form HTML is dead immediately after the first POST completes. On Android Chrome, some combination of (a) native tap event firing twice on flaky touch → two `click` events → two `submit` events, (b) Chrome's autofill "save password" flow re-triggering the form after the first submit, or (c) fast user re-taps because the visual feedback wasn't obvious enough — one of these fires a SECOND POST /login within ~1s of the first. That second POST carries the stale token from the same in-memory form → server has the new token in the fresh session → `hash_equals` fails → 403. **Diagnostic path**: (1) verified v0.7.3 was deployed on Bosco (`docker inspect` confirmed session.save_path pinned + named-volume mounted + 100 session files persisted); (2) grep container logs for 403 pattern → found `303 → 403` at 1s cadence; (3) cross-referenced Ayula access log with UA + referrer → all failing UAs were Android Chrome; (4) inspected karhu's `Csrf::validate` flow → confirmed session-backed storage rotates on regenerate; (5) inspected `AuthController::establishSession` → confirmed the regenerate happens on success; (6) inspected `templates/auth/login.twig` + `templates/layout.twig` → confirmed no existing anti-double-submit guard. **Fix**: single inline `<script>` added to `templates/layout.twig` (right after the v0.6.8 CSRF-refresh IIFE). A capture-phase `submit` listener on `document` sets `form.dataset.mishkaSubmitting = '1'` on the first submit, synchronously disables all `<button type="submit">` / `<input type="submit">` in the form (which stops the second native click from ever emitting a submit event on modern browsers), and `preventDefault()`s any submit event that still makes it through (belt-and-braces). Safety net: a 5s `setTimeout` clears the flag + re-enables buttons if the submit is cancelled by another script (real successful submits navigate away, making the timer irrelevant; the 5s is defence against corner-case listener interference). **Scope: EVERY form on the site.** Login is where the bug was visible (session/CSRF rotation makes the second POST 403), but every other form (chore-complete, event-create, household-invite, password-change, delete-account, invite-accept, chore-approve, list-add, etc.) was silently at risk of double-writes on a mobile double-tap. **Why capture phase**: our listener runs BEFORE any other submit handler could `preventDefault` or `stopPropagation` on the event; the flag lands unconditionally on submit #1 so submit #2's check is authoritative. **Why not `stopImmediatePropagation` on submit #2**: intentionally excluded — other listeners on the same form should still see the event (even if we cancelled it) so any bookkeeping they do isn't skewed by our intervention. **Why not `form.submit()` guard**: JS `form.submit()` bypasses the submit event by spec (per WHATWG); no mishka code calls `form.submit()` from JS today, so out of scope. If added later, that call site would need its own guard. **Why not server-side**: the karhu-Csrf-level fix would be either (a) grace-window acceptance of the immediate-prior token for N seconds after regenerate, or (b) skip CSRF validation on `/login` when the session is already authed (redirect instead). (a) is a karhu framework change (out of scope + karhu is parked at v0.1.0); (b) requires middleware-order changes because Csrf runs before dispatch. Client-side prevention is O(1) code + zero framework churn + also fixes silent-double-write on all other forms. **Why not disable-and-forget**: the 5s safety net matters for auto-fill scripts or password managers that call `preventDefault` after the browser has fired submit (rare on modern Chrome but observed on some Android WebView contexts). Without the timeout, a legitimately-cancelled submit locks the form permanently. **What v0.7.3 did NOT solve that this does**: v0.7.3 fixed "your session file was wiped by a container recreate" — the class where a user's stale PHPSESSID cookie found no server-side session at all. v0.7.4 fixes "your session rotated MID-USER-INTERACTION because you successfully logged in, but the browser re-fired the form" — the class where the session is fine but the CSRF token in the form is one rotation stale. Both are real; both had to be fixed; the fix for each is different. **What v0.7.4 does NOT solve**: (a) a stale CSRF token in HTML cached by SW from BEFORE a deploy — mitigated by v0.6.8's inline `/csrf-token` refresh script (fetches on DOMContentLoaded + pageshow(persisted)); (b) a cross-tab session rotation where tab A logs in but tab B has stale form HTML — same mitigation, plus the tab B's next form submit is a legitimate user action that should probably notify them that their session changed; (c) session expiry mid-form-fill after `gc_maxlifetime` — deferred candidate from #65 (bump gc_maxlifetime + cookie_lifetime for family-app "stay logged in"). **Local validation**: `composer test` (762/1885/0 unchanged — no PHP changes); local `docker build` + `curl /login` returned 200 with the new inline script present in the HTML source. No new test asserting the inline script itself — the guard is client-side JS with no PHP-testable surface, and adding a template smoke test for a single fragment is overkill for a hotfix. **Deployed as fast-path** (no plan-mode, no 4-round-review) per user approval — scope is narrow (1 file: layout.twig, plus SW/README/DOCS), verified locally, well-known pattern. **SW_VERSION bump to mishka-v0.7.4** per decision #51 always-bump (layout.twig changed → the SW-cached HTML for every route needs to invalidate on next nav).

65. **Persistent PHP session storage** (v0.7.3 hotfix). Mobile + tablet users started reporting CSRF-token-mismatch 403s after the v0.7.2 deploy. Investigation: the v0.7.2 self-contained image inherited PHP's default `session.save_path = /tmp`, which is container-ephemeral. Every `docker compose up -d` — including the v0.7.2 deploy itself, 5 hours before this hotfix — wiped every session file. Mobile/tablet users with long-lived tabs (backgrounded, then foregrounded via home-screen icon or after >24 min inactivity) held: (a) a stale `PHPSESSID` cookie whose server-side file was gone; (b) a stale CSRF token in the `<meta name="csrf-token">` and `<input name="_csrf_token">` embedded in the HTML from the pre-deploy render. On POST: `session_start()` initialised an empty `$_SESSION` (use_strict_mode=0 lets PHP re-use the client's PHPSESSID); karhu's `Csrf::getStoredToken()` returned `''` (no session-side token AND `_csrf_cookie` fallback was dead code — `Csrf::regenerate()` never sets it); `hash_equals('', form_token)` failed → 403. The inline `/csrf-token` refresh script (DOCS #49, v0.6.8) DOES try to fix this on `DOMContentLoaded` + `pageshow(persisted)`, but doesn't fire on home-screen resume or on a tab that was already open before the deploy. Diagnostic numbers: 5 CSRF-mismatch 403s / 10 total login POSTs in 6 hours (50% failure). **Fix**: Dockerfile adds `RUN mkdir -p /var/lib/mishka/sessions && chmod 733 ... && echo 'session.save_path = "/var/lib/mishka/sessions"' > /usr/local/etc/php/conf.d/mishka-sessions.ini`. `docker-compose.yml` adds a named volume `mishka-sessions:/var/lib/mishka/sessions` mounted on both `app` and `worker` services (worker doesn't use `$_SESSION` but mounting keeps the image's directory contract intact). Session files now survive container recreates — mobile clients with old PHPSESSID find their session on disk with the real `_csrf_token`, CSRF validates. **Not baked into the image**: sessions are runtime state, not code; the named volume is the correct docker primitive. The image is still "self-contained" for code + vendor (decision #64); the added state volume mirrors how `/data/mishka/logs` was already a runtime concern outside the image. **`chmod 733`**: the container runs as root (no USER directive), so root-writable is sufficient today, but 733 is future-safe if we ever switch to a non-root user (v1.0+ candidate — DOCS #64). **NOT included in this hotfix** (deferred to v0.7.4+ if needed): `session.gc_maxlifetime` bump from 1440s (24 min) to something family-app-appropriate like 30 days; `session.cookie_lifetime` bump from `0` (browser-session) to a fixed duration so tabs survive browser restarts. These are UX improvements on top of the correctness fix landed here; safe because v0.7.0's `/me/sessions` UI already gives users per-device revoke. **Verified end-to-end locally**: `docker build` succeeded; `docker run --env-file .env -v mishka-sessions-test:/var/lib/mishka/sessions mishka-php:v0.7.3-test` → `curl /login` set `PHPSESSID=c7af93d…`; `docker exec ... ls -la /var/lib/mishka/sessions/` showed `sess_c7af93d…` inside the volume; after `docker stop`, the volume STILL had the session file (`docker run --rm -v mishka-sessions-test:/vol alpine ls /vol` post-removal). Container recreate no longer wipes state. **Fast-path release** (no plan-mode + 4-round-review): the fix scope was narrow (2 files: Dockerfile + docker-compose.yml, plus SW + docs), config-only, empirically validated at build time. User approved fast-path. **SW_VERSION bump to mishka-v0.7.3** per decision #51 always-bump.

Discoverability via `/me/profile`'s new "🏆 Your badges" link + clickable badge emoji on the `/chores` leaderboard (each emoji wraps in `<a href="/badges">` — no nested anchors, the leaderboard `<li>` has no outer anchor). **Leaderboard source switch**: `ChoresController::achievementsBoard()` now sources the `badges` list-per-member from `BadgeAwardRepository::listByUserForHousehold()` (INNER JOIN household_members so departed members' awards stay in the table but are hidden from the leaderboard). Streak number still from Achievements. The merge uses `array_merge` (RIGHT-WINS) NOT `+` (LEFT-WINS) — PHP's `+` operator would silently keep the Achievements-computed `badges` field instead of overwriting (round-2 C6 gotcha). **NO push notification on badge earn** — scope deferral, v0.6.14+ candidate. **Departed-member badge history invariant**: users who left a household keep their `badge_awards` rows in the DB (household_id CASCADEs only on household-delete, not on member-leave). But `listByUserForHousehold` INNER JOINs `household_members` so the leaderboard + `/badges` roster hide departed-member awards. Same posture as decision #24's leaderboard departure semantics. **Migration ordering**: `bin/karhu migrate` must land badge_awards before `badges:backfill` runs; the backfill's INSERT would otherwise fail. Standard release-flow ordering already satisfies this. **SW_VERSION bump to mishka-v0.6.13** per decision #51 (always-bump) + decision #48 (form-shape change cost — new `/badges` page is a precached HTML shape).

## Development

```bash
composer install
cp .env.example .env
php vendor/bjornbasar/karhu/bin/karhu migrate
composer test           # PHPUnit (SQLite in-memory)
composer analyse        # PHPStan level 6
composer serve          # http://localhost:8080

# PG smoke locally against your real database:
DB_DSN='pgsql:host=…;dbname=…' DB_USER=… DB_PASS=… \
  vendor/bin/phpunit --filter 'PgSmoke'
```

CI runs two jobs: `test` (SQLite in-memory + PHPStan) and `pg-smoke` (postgres:16 service container; runs anything matching `PgSmoke`).
